# Построение единой модели высотности зданий
### Решение команды «Лунная призма, дай мне деняк» для Финала кейс-чемпионата Changellenge >> Cup IT 2026

## О проекте

Этот проект решает задачу объединения разнородных геоданных по зданиям Санкт-Петербурга и построения единой модели высотности зданий для последующего использования в задачах геоаналитики и планирования сети.

В рамках решения реализован ML-пайплайн, который:

- загружает и унифицирует данные из двух источников;
- очищает геометрию и атрибуты;
- сопоставляет объекты, относящиеся к одному физическому зданию;
- выбирает наиболее надежную известную высоту;
- предсказывает высоту для зданий, где она отсутствует;
- формирует итоговый слой зданий с финальным полем высоты.

## Контекст задачи

По условиям кейса необходимо объединить данные из двух источников геометрии зданий, очистить их, сопоставить объекты и получить единую качественную базу высотности, пригодную для задач планирования сети.

Ключевые особенности задачи:
- высота здания явно присутствует только в одном из источников;
- данные содержат пропуски, выбросы и геометрические несоответствия;
- одно физическое здание может быть представлено в разных источниках по-разному, включая сценарии «один ко многим» и «многие ко многим».

## Входные данные

Проект использует два исходных CSV-файла:

- `cup_it_example_src_A.csv`
- `cup_it_example_src_B.csv`

### Источник A
Содержит:
- идентификатор полигона;
- название объекта;
- теги;
- геометрию в формате WKT;
- площадь полигона;
- адрес из базы «Реформа ЖКХ»;
- минимальную и максимальную этажность.

### Источник B
Содержит:
- адресные признаки;
- назначение здания;
- этажность (`stairs`);
- среднюю высоту этажа;
- высоту здания (`height`);
- геометрию в формате WKT.

## Что делает пайплайн

После загрузки данных проект формирует единый набор зданий и рассчитывает итоговую высоту по следующей логике:

- если у здания есть надежная известная высота, сохраняется исходное значение;
- если высота отсутствует, модель предсказывает `predicted_height`;
- итоговое значение записывается в поле `final_height`.

## Архитектура решения

```text
Источник A + Источник B
        ↓
Унификация схемы и парсинг геометрии
        ↓
Очистка геометрий и атрибутов
        ↓
Пространственное сопоставление объектов
        ↓
Формирование единого слоя зданий
        ↓
Выбор надежной известной высоты
        ↓
Разделение на:
- train_eval_set (высота известна)
- predict_set (высота неизвестна)
        ↓
Feature engineering
        ↓
Spatial GroupKFold
        ↓
Сравнение моделей:
- LightGBM
- CatBoost
- baseline-подходы
        ↓
Обучение лучшей модели на всех известных высотах
        ↓
Предсказание высоты для predict_set
        ↓
Экспорт итогового результата
```

## Основные этапы обработки

### 1. Загрузка и унификация
Источники читаются и приводятся к общей структуре для дальнейшей совместной обработки.

### 2. Очистка данных
На этом этапе выполняются:
- парсинг WKT/geometry;
- удаление пустых и некорректных геометрий;
- фильтрация нерелевантных объектов по площади;
- приведение к единой системе координат.

### 3. Пространственное сопоставление
Объекты из разных источников сопоставляются по пространственному пересечению и IoU, чтобы определить, какие записи относятся к одному и тому же зданию.

### 4. Выбор высоты
Если для здания есть несколько кандидатов по высоте, применяется набор правил для выбора наиболее надежного значения.

### 5. Формирование обучающей и предсказательной выборки
Создаются две выборки:
- `train_eval_set` — здания с известной высотой;
- `predict_set` — здания, для которых высоту необходимо оценить моделью.

### 6. Feature engineering
Используются признаки двух типов.

**Безопасные до разбиения:**
- площадь;
- периметр;
- компактность;
- координаты центроида;
- размеры bounding box;
- прямоугольность;
- вытянутость.

**Строящиеся только на train fold:**
- grid statistics;
- KNN-признаки;
- priors по типу здания;
- кодирование категориальных признаков.

### 7. Обучение и валидация
Сравниваются несколько подходов:
- `LightGBM`
- `CatBoost`
- baseline-модели

Оценка выполняется через **spatial cross-validation**, чтобы модель проверялась на пространственно отделенных областях.

### 8. Финальное обучение
После OOF-оценки лучшая модель обучается на всех объектах с известной высотой и применяется к `predict_set`.

### 9. Экспорт результата
Итоговый слой сохраняется в формате GeoPackage:

```bash
spb_buildings_final.gpkg
```

## Защита от data leakage

В проекте отдельно реализованы меры против утечки информации:

- `predict_set` не участвует в обучении и валидации;
- используется `Spatial GroupKFold`, а не случайное разбиение;
- статистики, завязанные на целевую переменную, считаются только на `train_fold`;
- KNN-признаки для train строятся в режиме leave-one-out;
- признаки с прямой утечкой исключаются из обучения.

### Примеры устраненных источников leakage
- self-lookup в KNN;
- расчет grid statistics на всем датасете до split;
- расчет priors по типу здания на полном наборе данных;
- случайное разбиение соседних объектов в разные выборки;
- использование `stairs` как почти прямого proxy для target.

## Метрики качества

Для оценки качества используются:

- `MAE`
- `RMSE`
- `R²`
- `MAPE`
- `MedAE`
- `P90`

Это позволяет оценивать модель не только по средней ошибке, но и по устойчивости к выбросам и качеству работы на сложных объектах.

## Результат

На выходе формируется итоговый слой зданий с основными полями:

- `building_id`
- `height`
- `predicted_height`
- `final_height`
- `height_source`
- `area_m2`
- `centroid_lat`
- `centroid_lon`
- `bld_type`
- `geometry`

### Логика итоговой высоты
- если `height` известна, используется она;
- если `height` отсутствует, используется `predicted_height`;
- итог записывается в `final_height`.
